# 适用于ROS1的IMU驱动
一款解决串口缓冲区延迟导致时间戳同步问题的**高精度ROS1 IMU驱动**。

## 存在问题
通过串口读取IMU数据时，操作系统的串口缓冲区会引入不可预测的延迟。这会导致ROS节点接收到数据时的时间戳，与IMU实际生成数据的时间不匹配，最终造成IMU消息时间戳不准确。

## 解决思路
本驱动实现了**线性时间同步算法**，建立并持续更新以下两者的线性映射关系：
- IMU内部时钟（`cnt1ms` - 毫秒计数器）
- ROS系统时间

算法采用**滑动窗口增量最小二乘法**，实现以下功能：
1. 实时拟合线性映射公式：`ROS时间 = a × IMU时间 + b`
2. 自动剔除异常值（与预测值偏差大于50ms）
3. 指数平滑滤波（0.9/0.1）稳定映射参数
4. 为一批数据中的所有帧回溯精准时间戳

## 核心特性
- **时间戳高精度**：采用IMU自带1ms精度内部时钟，替代系统接收时间
- **在线自校准**：自适应IMU与主机之间的时钟漂移
- **异常值过滤**：自动剔除时间戳异常数据
- **批量数据处理**：支持单次串口读取多帧IMU数据
- **数据完整性校验**：检查帧头(0x55 0x55)、帧尾(0xBF)与校验和

## 数据帧格式
驱动适配**134字节**固定长度IMU数据帧，结构如下：

| 字段 | 偏移 | 长度 | 说明 |
|------|------|------|------|
| 帧头 | 0-1 | 2字节 | 固定为0x55 0x55 |
| 毫秒计数器 | 2-5 | 4字节 | IMU内部cnt1ms时间戳 |
| 陀螺仪数据 | 6-17 | 12字节 | 原始int32格式三轴陀螺 |
| 加速度计数据 | 18-29 | 12字节 | 原始int32格式三轴加速度 |
| 温度 | 30-31 | 2字节 | 器件温度 |
| 其他导航数据 | ... | ... | 预留扩展字段 |
| 校验和 | 132 | 1字节 | 第2~131字节累加和 |
| 帧尾 | 133 | 1字节 | 固定为0xBF |

## 使用方法
1. 将IMU连接至 `/dev/ttyUSB0`（可在源码中修改端口）
2. 启动驱动节点：
   ```bash
   roslaunch imu_driver imu_driver.launch
   ```
3. 查看IMU数据：
   ```bash
   rostopic echo /imu/data
   ```

## 发布话题
- `/imu/data`（[sensor_msgs/Imu](http://docs.ros.org/api/sensor_msgs/html/msg/Imu.html)）
  - `header.stamp`：通过IMU内部时钟计算的精准时间戳
  - `angular_velocity`：陀螺仪数据（单位：rad/s）
  - `linear_acceleration`：加速度计数据（单位：m/s²）

## 可配置参数
可直接在 `src/imu_driver.cpp` 中修改以下参数：
- 串口设备：第281行（默认 `/dev/ttyUSB0`）
- 波特率：第281行（默认 115200）
- 时间同步窗口：第73行（默认 `WINDOW_SIZE = 1000`）
- 异常值阈值：第87行（默认 0.05秒 = 50ms）
- 平滑系数：第143-144行（默认 0.9 / 0.1）

## 算法说明
### 时间同步原理
`TimeSync` 类维护1000组时间戳样本 `(IMU时间, ROS时间)`，并执行：
1. **增量最小二乘**：高效计算线性回归，无需保存全部历史数据
2. **异常值剔除**：丢弃预测误差超过50ms的样本
3. **指数平滑**：使用 `a = 0.9×旧值 + 0.1×新值` 稳定参数

### 批量处理流程
当串口数据到达时：
1. 从缓冲区解析所有完整数据帧
2. 使用**最后一帧**更新时间同步模型
3. 用模型为这批所有帧计算精准时间戳
4. 发布带校正时间戳的IMU消息

即使因串口缓冲导致多帧数据同时到达，每帧也能获得**正确的历史时间戳**。
