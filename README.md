适用于ROS1的高精度IMU与GNSS驱动
一款解决串口缓冲区延迟导致时间戳同步问题的高精度ROS1驱动，同时发布IMU和GNSS定位数据。

存在问题
通过串口读取IMU/GNSS数据时，操作系统的串口缓冲区会引入不可预测的延迟。这会导致ROS节点接收到数据时的时间戳，与传感器实际生成数据的时间不匹配，最终造成时间戳不准确。

解决思路
本驱动实现了线性时间同步算法，建立并持续更新以下两者的线性映射关系：
传感器内部时钟（cnt1ms - 毫秒计数器）
ROS系统时间

算法采用滑动窗口增量最小二乘法，实现以下功能：
实时拟合线性映射公式：ROS时间 = a × IMU时间 + b
自动剔除异常值（与预测值偏差大于50ms）
指数平滑滤波（0.9/0.1）稳定映射参数
为一批数据中的所有帧（包括IMU和GNSS）回溯精准时间戳

核心特性
时间戳高精度：采用传感器自带1ms精度内部时钟，替代系统接收时间
在线自校准：自适应传感器与主机之间的时钟漂移
异常值过滤：自动剔除时间戳异常数据
批量数据处理：支持单次串口读取多帧数据
数据完整性校验：检查帧头(0x55 0x55)、帧尾(0xBF)与校验和
双传感器融合输出：同步发布IMU原始数据与GNSS定位数据

数据帧格式
驱动适配134字节固定长度数据帧，结构如下：
字段   偏移   长度   说明
帧头   0-1   2字节   固定为0x55 0x55

毫秒计数器   2-5   4字节   内部cnt1ms时间戳

陀螺仪数据   6-17   12字节   原始int32格式三轴陀螺

加速度计数据   18-29   12字节   原始int32格式三轴加速度

温度   60-61   2字节   器件温度

GNSS纬度   62-69   8字节   单精度浮点数，单位：度

GNSS经度   70-77   8字节   单精度浮点数，单位：度

GNSS高度   78-81   4字节   相对高度，单位：mm

东向速度   82-85   4字节   单位：mm/s

北向速度   86-89   4字节   单位：mm/s

天向速度   90-93   4字节   单位：mm/s

其他导航数据   ...   ...   预留扩展字段

校验和   132   1字节   第2~131字节累加和

帧尾   133   1字节   固定为0xBF

使用方法
将设备连接至 /dev/ttyUSB0（可在源码中修改端口）
启动驱动节点：
   bash
   roslaunch imu_driver imu_driver.launch
   查看数据：
   bash
   # 查看IMU数据
   rostopic echo /imu/data
   # 查看GNSS数据
   rostopic echo /gnss/fix
   
发布话题
/imu/data (sensor_msgs/Imu)
  header.stamp：通过内部时钟计算的精准时间戳
  angular_velocity：陀螺仪数据（单位：rad/s）
  linear_acceleration：加速度计数据（单位：m/s²）

/gnss/fix (sensor_msgs/NavSatFix)
  header.stamp：与IMU严格对齐的时间戳
  latitude / longitude：经纬度信息
  altitude：海拔高度（单位：米）
  position_covariance：位置协方差（对角线已知）

可配置参数
可直接在 src/imu_driver.cpp 中修改以下参数：
串口设备：第281行（默认 /dev/ttyUSB0）
波特率：第281行（默认 115200）
时间同步窗口：第73行（默认 WINDOW_SIZE = 1000）
异常值阈值：第87行（默认 0.05秒 = 50ms）
平滑系数：第143-144行（默认 0.9 / 0.1）

算法说明
时间同步原理
TimeSync 类维护1000组时间戳样本 (IMU时间, ROS时间)，并执行：
增量最小二乘：高效计算线性回归，无需保存全部历史数据。
异常值剔除：丢弃预测误差超过50ms的样本。
指数平滑：使用 a = 0.9×旧值 + 0.1×新值 稳定参数。

批量处理与同步发布
当串口数据到达时：
从缓冲区解析所有完整数据帧。
使用最后一帧更新时间同步模型。
用模型为这批所有帧计算精准时间戳。
同时发布带校正时间戳的IMU消息和GNSS消息，确保两类数据在时间轴上严格对齐。

即使因串口缓冲导致多帧数据同时到达，每帧也能获得正确的历史时间戳。
